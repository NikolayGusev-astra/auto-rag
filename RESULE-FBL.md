## 0. Вердикт (TL;DR)

Репозиторий — это **амбициозная и местами грамотно спроектированная архитектура, в которой значительная часть заявленных фич не работает, не подключена или сломана молча**. Корневая причина большинства проблем — идиома `except Exception: pass`, которая позволила трём `NameError` в продакшн-путях (веб-fallback, полнотекстовое извлечение, федерация) жить незамеченными, при этом один из них прикрыт «зелёным» регрессионным тестом, который баг не покрывает.

Три системных вывода:

1. **Документация продаёт систему, которой нет.** «LLM verify» из README — мёртвый код. «Trafilatura full-text» — `NameError` при каждом вызове. «patch_dcd() retrains router» — regex-патчер, чья позитивная половина не матчится с реальным форматом файла. «Auto-rollback if regression >5%» — это `print()` с подсказкой команды `cp`.
2. **Контур самопроверки системы сломан ровно в тех местах, где он нужен.** Canary-деплой сравнивает baseline с самим собой (и при этом затирает ваши изменения), калибровка судьи калибрует не тот промпт не на том домене на n=12, golden set из 12 вопросов статистически не может отловить регрессию 5%, а каждое попадание memvid-кэша оценивается eval-харнессом как **ноль**.
3. **Концептуальные решения не выдерживают челленджа** (подробно в §7): keyword-роутинг со сломанной нормализацией confidence, семантический кэш без инвалидации и калибровки порога, безусловная утечка каждого запроса в веб при «local-first» позиционировании, три параллельных пайплайна и три eval-харнесса.

При этом есть и настоящая инженерия: fail-open деградация memvid, атомарная миграция капсул, трассировка RagTrace, честные unit-тесты чистых функций, `memvid_canary.py` как единственный серьёзный деплой-инструмент. Скелет хороший — начинка требует ремонта.

## 3. Как система работает НА САМОМ ДЕЛЕ

Реальный поток `async_rag_search` (`rag_async.py:669-743`):

```
1. memvid.recall (ДО кэша!) — если score ≥ 0.75 → вернуть прежний ответ
   (другая форма результата: без "chunks", sources-список → ломает потребителей, §5.3)
2. Compound-детект по подстрокам → fan-out на 2 подзапроса в фантомный домен
   "rusbitech" (в роутере не существует) → конкатенация чанков без дедупа/реранка
3. _async_rag_search_impl:
   a. LRU-кэш (md5(query|domain), 100 записей, БЕЗ TTL)
   b. ZVec ∥ SearXNG-web — ВСЕГДА параллельно, web на каждый запрос
   c. entity-гейт (regex) → при несовпадении MCP-цепочка → web
   d. score-гейт: max_score ≥ 0.75 (LLM_EVAL_HIGH_THRESHOLD — имя про LLM,
      используется как cosine-порог; bge-m3 редко даёт 0.75 → ветка почти недостижима)
   e. confidence < 0.20 → MCP → web → пусто (срабатывает для 9/12 golden-запросов,
      т.к. нормализация confidence сломана, §5.2)
   f. финальный MCP → web → федерация (при непустом ответе — NameError по `k`, §5.1)
4. memvid.record при промахе (compound-результаты НЕ записываются никогда)
```

Ключевое: **DCD-роутинг вычисляет collection, которую retrieval игнорирует.** `_blocking_zvec` захардкожен на `category = 'wiki' OR 'llm-wiki'` (`rag_async.py:327`), zvec_server — аналогично (`zvec_server.py:102-106`), а индексаторы выводят category из имени каталога (`indexer.py:289`), которое никогда не совпадёт с доменами DCD. 400-строчная таблица keywords производит декоративный выход; реально используется только `confidence` — который сломан. `primary_source`/`fallback_sources` — самая проработанная часть конфига доменов — **не читаются нигде** (grep: 0 использований в пайплайне).

### 4.3 HIGH/MAJOR — корректность и надёжность

- **Инкрементальная индексация никогда не удаляет чанки** изменённых/удалённых файлов (doc ID от хеша контента → правка = новые ID, старые живут вечно). Начатая логика удаления (`tracked = set(state.keys())`) брошена. Инкрементальный режим монотонно деградирует качество retrieval. — `indexer.py:248-262,251`; `zvec_incremental_indexer.py:226-239`.
- **Файлы помечаются «done» даже при сбое вставки** — упавшие батчи навсегда пропускаются при следующих `--incremental`. — `indexer.py:348-368`; `zvec_incremental_indexer.py:325-347`.
- **Zero-vector-отравление**: четыре хелпера при сбое эмбеддинга молча подставляют `[0.0]*1024`; zvec_server выполняет поиск по нулевому вектору с «нормальными» скорами, ChromaIndexer **вставляет** нулевые векторы в индекс навсегда. — `unified_searcher.py:50`, `chroma_adapter.py:46,204-210`, `zvec_server.py:50`, `zvec_adapter.py:95`.
- **ChromaIndexer обрезает каждый файл до 2000 символов** (никакого чанкинга) и может молча потерять последний батч. — `chroma_adapter.py:192,203,184-185`.
- **«Persistent tunnels» федерации — миф**: на каждый запрос создаётся новый клиент, SSH-туннель (fix 2s sleep, случайный порт 18000-19000 без ретрая коллизий) и рвётся в `finally`; существующий синглтон не используется. — `rag_federated.py:410-420,167,185,389-397`.
- **`memvid_canary run --quick` не может работать**: передаёт `--quick` в eval_golden, чей argparse его не знает → exit 2 → RuntimeError. Рекламируемый режим никогда не запускался. — `memvid_canary.py:317-318` vs `eval_golden.py:271-278`.
- **rag_reindex сам себя обманывает**: пишет LOCK внутрь коллекции при каждой проверке → при запуске чаще раза в сутки staleness никогда не детектится; `doc_count` = число файлов индекса минус 1, а не документов. — `rag_reindex.py:94-102`.
- `_blocking_web` обогащает полным текстом максимум **одну** страницу (`len(chunks) < 1`), недостижимый `return []`. — `rag_async.py:372,392`.
- Кэш: compound-результаты пишутся в кэш, который для них **никогда не читается**; кэш отдаёт разделяемый мутабельный dict; TTL нет — «пустой» результат живёт до вытеснения даже после переиндексации. — `rag_async.py:733,521-524,236`.
- Blanket `except Exception: pass` — минимум 9 мест в горячих путях (`rag_async.py:314,337,385,390,422` и др.) — корневая причина «молчаливо сломанного» класса багов.

### 4.4 DCD — систематические ошибки классификации (все проверены запуском)

- **Русский язык фактически не поддерживается**: 14 из 15 доменов не имеют ни одного кириллического keyword; кириллица есть только у... **ford-club**. Итог: «настроить спам фильтр для почты» → **ford-club** (через «фильтр»), «заменить диск в рейде» → **ford-club** (через «диск»), «как настроить репликацию базы данных» → software-dev с confidence 0.000. Для системы с русскими промптами, русским golden set и русской вики это худший практический дефект роутинга. — `dcd_router.py:279-285`.
- Стоп-слова как взвешенные keywords: «at», «if», «case», «go», «free», «top», «access», «key», «ip» → «best pizza recipe with go» → software-dev. — `dcd_router.py:23,26,84,234` и др.
- Anti-keywords — компаундирующее **вето**, а не штраф: `score *= 0.3` за каждый; 3 anti-хита = −97.3%. Легитимный запрос «docker nginx letsencrypt on systemd» аннигилирует linux-admin. — `dcd_router.py:333-335`.
- Мёртвые keywords: «ci/cd», «802.1q» не могут заматчиться никогда (нормализация удаляет `/` и `.`). — `dcd_router.py:298,324-325`.
- Дубликаты ключей словаря молча теряют веса («argocd» 4→3), hardware — копипаста хвоста storage → tie-break по порядку словаря. — `dcd_router.py:48,51,68,72,109,111,264-267`.
- Биграммный re-join уничтожает порядок слов: «compose docker» матчит keyword «docker compose». — `dcd_router.py:313`.
- «Learning loop» (§7.3): циркулярная метка (actual_source зависит от самого роутинга), `has_content` читается и не используется, `confidence` предложения = доля пары ошибок среди всех ошибок (не уверенность), «авто-патч при >0.8» из докстринга не реализован — `--apply` применяет всё безусловно, SQL-поля ссылаются на несуществующие таблицы. — `dcd_learner.py:294-309,298,181,339-341,182-183`.

### 4.5 memvid — полный список в §7.2; кратко

- Порог 0.75 сравнивает запрос не с прошлым **запросом**, а с эмбеддингом до-6КБ-блоба чанков (асимметричный query↔document скор); эпизоды без ответа индексируются по эмбеддингу запроса — **два разных векторных пространства в одном индексе**. — `memvid_memory.py:330-331,372`.
- Семантика скора SDK не валидируется нигде (cosine? [0,1]? higher-better?). — `memvid_memory.py:397`.
- `record()` возвращает True когда ничего не сохранено (эмбеддер упал → put=None → commit → True). — `memvid_memory.py:332-334,665-676`.
- RLock удерживается через 10-секундный HTTP-вызов эмбеддера → head-of-line blocking на «быстром» пути. — `memvid_memory.py:452,469,624,659`.
- Домен-фильтр — post-filter по глобальному top-k (может вернуть [] при существующих совпадениях); эпизоды с domain=None протекают во все домены. — `memvid_memory.py:639-640`.
- Смена embedding-модели молча инвалидирует индекс: `embedding_identity` пишется, но не проверяется при открытии; ошибки recall гасятся до debug-лога. — `memvid_memory.py:346-350,403-405`.
- Нет межпроцессной блокировки капсулы; canary **удаляет живые файлы капсулы посреди прогона**. — `memvid_canary.py:350-352`.
- Дубликаты эпизодов накапливаются без дедупа; каждый прогон eval с memvid дописывает весь golden set заново. — `rag_async.py:739-741`.
- CLI эмбеддит запросы, минуя `bridge_memvid_env()` → может искать не той моделью, которой записано. — `hermes_memory_cli.py:59-68` vs `rag_async.py:101-102`.
- Memory-хит **роняет CLI**: `result['sources'].items()` на списке → AttributeError. — `rag_async.py:762-767`.

## 5. Челлендж концепции — главный раздел

### 7.1 «Local-first», который отправляет каждый запрос в интернет

ZVec и SearXNG запускаются **параллельно на каждый некэшированный запрос** (`rag_async.py:534-538`), независимо от confidence и от того, что ZVec ответит с 0.9. Для системы, позиционируемой как «local-first» для **внутренних** вики/Jira (включая названия внутренних продуктов и тикеты), это приватностная дыра by design: `SEARXNG_ENABLED` по умолчанию true (`rag_config.py:109`). При этом latency-выгода почти нулевая: в entity-mismatch-ветке веб-результаты откладываются до провала всей MCP-цепочки (`rag_async.py:554-565`). Спекулятивный веб-поиск должен быть как минимум гейтирован DCD-confidence/ZVec-скором, а лучше — opt-in per domain.

### 7.2 Эпизодическая память (memvid): кэш «навсегда» без калибровки, инвалидации и обратной связи

Это концептуально самая опасная часть системы. По пунктам:

1. **Порог 0.75 не обоснован ничем.** Он существует как env-default и строка README. Ни калибровочного скрипта (при том что для судьи `calibrate_judge.py` есть!), ни precision/recall-кривой, ни теста граничного поведения, ни логирования около-пороговых решений. Хуже: инструмент, который мог бы его откалибровать (memvid_canary + eval), сломан так, что каждый хит = 0.0 (§4.1 C6) — любая попытка калибровки скажет «выключите кэш».
2. **Это не «тот же вопрос», это «похожая тема».** Скор — асимметричное сходство запроса с 6КБ-блобом чанков. Дense-эмбеддинги систематически размывают именно то, чем различаются технические вопросы: отрицания («включить» vs «выключить»), версии («Astra 1.7» vs «1.8»), значения параметров. False positive возвращает **полный чужой ответ как ответ**, с источниками, урезанными до голых имён систем (`rag_async.py:137-141`), и флагом `from_memory`, который ни один потребитель не показывает.
3. **Write-once, serve-forever.** Нет TTL, нет проверки `created_at` при recall, нет связи с переиндексацией (`indexer.py --clear` не трогает капсулу). Запись — только при промахе → попавшая в кэш запись **не может быть обновлена штатной работой** никогда. После переиндексации память возвращает дословный контент, которого больше нет в корпусе. Единственные средства — `compact`/`purge` — дописывают вместо удаления (C8).
4. **Кэшируются непроверенные ответы.** Записывается всё, что пайплайн выдал на промахе — включая веб-сниппеты со сфабрикованным скором 0.6 и MCP-выхлоп 0.7 — без какого-либо качественного гейта. В коде **уже есть** `_llm_verify` и поле `feedback: correct/wrong` (`memvid_memory.py:114`) — оба не используются: эпизод, помеченный «wrong», продолжал бы отдаваться вечно. Один неправильный ответ, записанный однажды, повторяется дословно для всей своей семантической окрестности.
5. **Нет сигнала использования.** Хит не инкрементирует счётчик, не пишет запись — система не отличает эпизод, отданный 1000 раз, от никогда не отданного. Даже рабочему compact нечем «выбрать нужные».
6. **Зависимость memvid-sdk не оправдана.** Реально используется: create/use, put_many(embeddings), ask(query_embedding), commit, stats — то есть **таблица с векторной колонкой**. Всё «фирменное» либо мёртво (temporal), либо фикция (шифрование), либо нерелевантно (видео-кодирование). История интеграции — обвинительное заключение: MEMVID-001 (неверное имя модуля → молчаливый полный noop), sidecar-костыль `.vecidx.jsonl` из-за того, что SDK не индексировал, миграционная машинерия, `_iter_frames`, пробующий 4 имени метода (`hermes_memory_cli.py:73-81`), и `_native_episodes`, восстанавливающий payload'ы **лексическим поиском по собственным заголовкам** O(n) (`hermes_memory_cli.py:101-137`), потому что SDK не умеет читать документы обратно. SQLite + sqlite-vec/numpy (тысячи эпизодов ×1024 float — брутфорс-косинус за субмиллисекунды) удалил бы половину `memvid_memory.py` и весь восстановительный код. Формат выбран по названию на коробке, а не по требованиям.
7. **Честная альтернатива уже написана и не включена**: `recall_as_context` / Option B в `integration_patch.py:71-75` — память как **контекст промпта** (augment, not replace), где ошибочный prior разбавляется свежим retrieval. Включён агрессивный Option A (short-circuit).

**Вывод:** идея эпизодической памяти — здравая, реализация short-circuit-кэша без калибровки/инвалидации/фидбэка — production-hazard. Достойный вариант: augmentation-режим + LLM-verify перед любым short-circuit + TTL + инвалидация по переиндексации.

### 7.3 «Learning loop» DCD — это не обучение

По README — «patch_dcd() retrains router». По факту: (а) детекция мисраутов на **циркулярной прокси-метке** — `actual_source` сам является функцией решения роутера и доступности источников, ground-truth нет вообще; (б) частотный подсчёт слов запросов; (в) regex-патчер исходника, чья позитивная половина не матчится с файлом (silent no-op), а негативная накапливает компаундирующие 0.3^n-штрафы без дедупа при повторных `--apply`. Нет целевой функции, нет train/val-разбиения, нет прогона eval до/после патча (харнесс существует — learner его не зовёт), нет отката. Таксономия learner'а («rusbitech», «research», lodestone/confluence) принадлежит другому деплою и не существует в роутере. Это лог-анализ с самомодификацией кода вслепую, продаваемый как ML.

### 7.4 Keyword-роутинг против собственной инфраструктуры

Система **уже** держит embedding-сервис и локальный LLM — и при этом классифицирует домены ручными keyword-списками с измеренными хрестоматийными провалами: языковой разрыв (русский → 0.000 или ford-club), полисемия («go», «диск»), дрейф сопровождения (три параллельных копии списка доменов: `dcd_router.py:12`, `dcd_router_llm.py:28+147`, `dcd_learner.py:30` — уже разъехались). Centroid/kNN-классификатор поверх существующих эмбеддингов был бы языконезависим и согласован с retrieval-индексом почти бесплатно. Keyword-роутер оправдан как <1ms префильтр — здесь он единственный реально подключённый режим (`rag_async.py:43` импортирует напрямую keyword-classify; dcd_router_llm не подключён к пайплайну вообще).

И финальный удар: **даже если бы роутинг работал идеально — retrieval игнорирует его выход** (§3). Слой маршрутизации декоративен.

### 7.5 Контур качества не может измерить систему

- **n=12** вопросов golden set: 1 вопрос = 8.33пп accuracy; порог регрессии −5% **ниже гранулярности метрики**; биномиальный шум ±13пп SE. Обнаружение истинной регрессии 5пп требует сотен вопросов. Существующий `dataset/golden_set_lodestone.json` (25 вопросов) не подключён никуда.
- Golden set **тавтологичен**: запросы — мешки из собственных keywords роутера, `key_facts` — слова самого запроса → «LLM-судья по key_facts» вырождается в проверку эха. Нет негативов, нет out-of-domain, нет парафраз, один частично-русский вопрос при русском деплое. Один вопрос промаркирован неверно («linux-admin-1» → expected «database»), в тесте это названо «спорным» вместо исправления.
- **Судья**: qwen2.5-7b, max_tokens=10, один прогон, regex `0\.\d+|1\.0` — ответы «1», «0», «0,9» (русская запятая при русском промпте!) парсятся в 0.0. Fallback на keyword-match **только вверх** (LLM «incorrect» может быть повышен, LLM «correct» не перепроверяется) → систематическая инфляция accuracy. Судья видит только `chunks_text[:2000]`.
- **Калибровка судьи** калибрует **английский** промпт (прод — русский), на **12 само-написанных** мета-вопросах про RAG (прод-домен — Linux/PostgreSQL/ZFS), grid-search 19×19 порогов на n=12 — гарантированный оверфит; один элемент с verdict «weak» не может совпасть никогда (класса «weak» в маппинге нет) → потолок 11/12; результат — печатная рекомендация руками поправить константы, которые захардкожены.
- **Три несогласованных eval-харнесса**: eval_golden (12 вопросов, LLM-судья), run_golden (DCD-only, пустой expected_domain = автопасс, пишет в **тот же** файл отчёта, затирая полный eval), eval_golden_unified (собственные 24 запроса, substring-матчинг). Три определения «accuracy» в одном репо.
- Canary: C5 (baseline vs baseline + уничтожение кандидата) делает главный деплой-инструмент хуже чем бесполезным — он опасен. `memvid_canary.py` — единственный серьёзно сделанный (tolerant parsing, p50/p99, backup/restore с finally) — и тот с несуществующим флагом `--quick` и rollback-restore в файл **без расширения** (`memvid_canary.py:441`: `.mv2.bak` → suffix-манипуляция даёт `memory_tenant` без `.mv2`).

**Вывод:** система не может честно ответить на вопрос «стало лучше или хуже?». Все цифры качества, которые она о себе сообщает, — либо шум, либо систематически завышены, либо (для memvid) систематически занулены.

### 7.6 Fallback-цепочка и модель доверия

- Три несовпадающих порядка цепочки (README, config-константа, хардкод в коде). Публичный Context7 **раньше** внутренних Jira/Confluence — странный дефолт для энтерпрайз-ассистента; first-non-empty означает, что маргинальный публичный хит перекрывает богатый внутренний источник. Последовательный проброс с таймаутами по 15s — worst-case ~75s **после** уже выполненного веб-поиска, чей результат лежит неиспользованным.
- **Провенанс отсутствует**: чанкам приписываются выдуманные плоские скоры (MCP 0.7/0.9, web 0.6), федеративные чанки приходят со **само-заявленным** скором удалённого узла — и всё это сортируется в одну кучу и записывается в память как «ответы». Публичный сниппет может переранжировать локальную вики на голой константе. Единственные границы доверия — is_error-фильтр и (дырявый) SSRF-guard.
- Федерация RAG-узлов по SSH-туннелям — оправдываемая идея, **переинженеренная в текущем виде**: туннель на запрос (2s+ на установку), нет TLS/mTLS, один общий ключ с timing-unsafe сравнением, нет защиты от петель (S9). Persistent HTTPS/mTLS-клиент (или хотя бы использование уже написанного синглтона) проще и быстрее.

### 7.7 Пять LLM-ролей в одном десктопном приложении

Конфиг требует bge-m3 + bge-reranker + gemma-4-e4b + qwen2.5-7b + qwen3-4b одновременно резидентными в LM Studio (`rag_config.py:39-89`). Реальный fallback есть только у эмбеддингов; остальное деградирует в молчаливые no-op'ы (reranker → константа 0.5 с приклеенным фейковым `rerank_score`, неотличимым от настоящего). Порт 1234 захардкожен литералом в 6 файлах. Плюс докстринги/golden set говорят «qwen2.5-7b», а дефолт классификатора — gemma (`rag_config.py:50`). Реранкер — англоязычный ms-marco-MiniLM для преимущественно русских пар (скоры ≈ шум), причём конфиг-переменная модели игнорируется модулем (`local_reranker.py:25`).

### 7.8 Config «single source of truth» — фикция

`unified_searcher`, `chroma_adapter`, `zvec_server` объявляют собственные env-переменные с **другими именами** (`EMBEDDING_URL` vs `RAG_EMBEDDING_URL`) — задокументированные переменные на них не действуют. `chroma/rag_config.py` — разошедшийся форк (другая embedding-модель, другие пороги). `WEB_SEARCH_MAX_RESULTS` определён дважды — второе присваивание убивает env-переменную (`rag_config.py:105,211`). `.env.example` документирует переменные-пустышки (`RAG_EMBEDDING_DIM` не читается; e2e-тест выставляет `RAG_SEARXNG_ENABLED`, конфиг читает `RAG_SEARXNG`). Два state-файла двух индексаторов на одну коллекцию.

151 тест в 37 файлах; в голом окружении: 107 passed / 11 failed (все — от окружения: numpy/aiohttp/fastapi/yaml/LM Studio/DNS) / 5 skipped за 1.5s.

**Театр:**
- `test_golden_set_consistency.py:38-47` — буквальная тавтология: `if expected == actual: assert actual == expected`. Называется «test_domain_accuracy_stable», не может упасть в принципе.
- `test_federation_k_bug.py` — **ложно-зелёный регрессионный тест**: проверяет исправление, не исполняя сломанную строку (§4.1 C1). Самый опасный вид теста — создаёт уверенность в исправленности неисправленного.
- `test_memvid_wiring.py:62-69` — самонейтрализующийся: assert в форме «если A то X, иначе Y», где проходят оба исхода; сломанное сравнение порога останется зелёным.
- `test_canary_deploy.py:137-138` — `assert "PASS" in report or "REGRESSION" in report` (verdict всегда один из них); `test_memvid_smoke.py:29` — `assert bool in (True, False)`.
- `test_memvid_attr_cleanup.py` — assert'ы существования функций (и именно он держит «живым» мёртвый `_cosine`).

**Дыры покрытия ровно там, где живут реальные баги:** ноль тестов на eval_golden.llm_judge (regex/запятая), canary_deploy.main (C5 ловился бы одним tmpdir-тестом), memvid_canary --quick, hermes_memory_cli compact/purge (C8 ловился бы одним round-trip), rag_search.py целиком, zvec_adapter (rmtree-футган C4), zvec_server, streaming, полная маршрутная лестница rag_async. Все 5 содержательных memvid-тестов — `importorskip("memvid_sdk")` + требуют живой LM Studio → эффективное CI-покрытие семантического recall = 0; гарантия поставки — «noop-путь работает».

**Гигиена:** два модуля мутируют `os.environ` на import-time; двойная идентичность модулей (`rag_async` vs `rag_core.rag_async`) → два набора глобалов `_CACHE`/`_memory` в одной pytest-сессии; monkeypatch глобального `os.path.join`.
